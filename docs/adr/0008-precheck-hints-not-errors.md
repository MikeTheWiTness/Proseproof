# ADR-0008: PreCheck 仅标记不判定

- **状态**: 已采纳
- **日期**: 2026-07-02

## 背景

在 LLM 校对之前，可以对原文执行一些基于规则的检查（括号不成对、连续重复词、
连续重复标点等），以降低 LLM 遗漏这些问题的概率。

常规设计思路有两种：
- **自动判定**：规则发现异常后直接标记为 error/suggestion，跳过后面的 LLM
  校对（或将其作为已确认的错误合并到最终报告）。
- **仅标记提示**：规则发现异常后，仅生成一份"提示清单"注入 LLM 的校对提示词，
  由 LLM 自行判断是否为真正的错误。

两种思路的核心分歧在于对「规则准确性」的假设。

## 决策

**PreCheck 中间件仅标记位置和模式，不判定对错，不设置严重级别。**

理由如下：

1. **无法保证零误报**：任何看似格式错误的现象都可能是作者有意为之。一篇
   语言学论文在梳理常见错词病句时，自身就包含大量"错误"。规则引擎无法
   区分这种上下文。
2. **严重级别在逻辑上自相矛盾**：既然不确定是否是错误，给一个严重级别是
   没有意义的——你不能给"可能不是错误的东西"评严重程度。
3. **LLM 才是最终决策者**：PreCheck 的产出是"异常模式清单"而非"错误清单"。
   清单格式为结构化 JSON（位置 + 模式枚举值 + 匹配到的文本），不做判断，
   让 LLM 根据自己的语义理解决定是否报错。

PreCheck 仅覆盖「出现该模式后，与文本错误的高度统计相关性」的规则，如：
- 括号/引号不成对
- 连续重复标点（`。。`、`，，`）
- 连续重复词（≥2 字）
- 空片段 / 纯空白

对于存在大量合法反例的模式（如中英文间空格、全半角混用、`的/地/得` 混淆），
不在 PreCheck 中实现。

## 后果

**正面**:
- 杜绝了规则引擎误判导致的虚假错误报告。用户不会看到"AI 校对工具把论文中的
  例句标成了错误"这种尴尬情况。
- 分工清晰：规则负责"提示"（不遗漏），LLM 负责"判断"（不误报）。
- PreCheck 零 LLM 成本，只做字符串扫描，任何文档都可以无条件运行。

**负面**:
- 规则能发现的问题仍然需要 LLM 二次确认，无法跳过 LLM 调用。
- 提示清单可能干扰 LLM 的注意力——如果 LLM 过于关注提示位置而忽略其他
  错误，反而降低了校对质量。需要在 prompt 中明确提示的辅助性质。
- 覆盖的规则类型受限（只做高置信度模式），大量潜在问题仍需 LLM 自行发现。
