# ADR-0003: 校对数据结构

- **状态**: 已采纳
- **日期**: 2026-07-02

## 背景

校对产出的核心数据是"修改记录"——LLM 发现的问题及其修改建议。当前的定位方式使用 `【N|原文|改为】` 内联标记（N 为序号），已在实际使用中验证可行。但随着校对场景的扩展，下游排版阶段需要更丰富的元数据来做出差异化呈现。例如，严重的语法错误需要红色高亮，而风格建议只需灰色下划线。纯粹的内联标记无法携带这种分类信息。

## 决策

在现有 `【N|原文|改为】` 内联标记的基础上，增加结构化字段：

**新增字段**:
- `type`: 枚举值 `error`（错误）或 `suggestion`（建议）。区分必须修改的问题和可选的优化建议。
- `severity`: 枚举值 `critical`（严重）、`major`（主要）、`minor`（次要）、`info`（提示）。表示问题的严重程度。

**LLM 声明方式**:
LLM 在校对输出的"修改原因"字段中，使用 `[type|severity]` 前缀声明分类信息。例如：
```
[error|critical] 主语和谓语不一致，会导致读者误解句意。
[suggestion|minor] 建议使用更正式的措辞。
```

**定位方式保持不变**: 沿用 `【N|原文|改为】` 内联标记，确保向后兼容。

## 后果

**正面**:
- 下游（LaTeX 排版）可按严重程度和类型做差异化呈现（颜色、标注样式等）。
- 数据分析可按 type/severity 维度统计校对质量。
- 不破坏现有的内联标记格式，与已有流程兼容。

**负面**:
- 依赖 LLM 正确输出 `[type|severity]` 前缀，可能存在标注不准确的情况。
- 严重程度判断存在主观性，不同 Profile 可能需要不同的判断标准。
- 增加了数据结构复杂度，解析阶段需额外处理前缀不匹配的容错逻辑。
