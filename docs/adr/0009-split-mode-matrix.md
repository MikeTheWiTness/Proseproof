# ADR-0009: 分割模式矩阵

- **状态**: 已采纳
- **日期**: 2026-07-02

## 背景

Proseproof v0.1.0 继承自 JiaoDuiAgent，包含六种分割模式：`rule`（正则切分）、
`none`（不切分）、`smart`（LLM 全文 `<problem>` 标签）、`manual`（人工标记）、
`knowledge_smart` 和 `knowledge_manual`（K-12 知识文档专用）。

在通用化过程中需要重新审视：

1. `knowledge_*` 模式高度耦合于 K-12 的"知识点"概念，不应带入通用版本。
2. `smart`（全文 LLM）成本高，且在大纲驱动的 `smart-flash` 出现后，定位重叠。
3. `rule`（正则）在通用场景中可靠性取决于用户是否理解自己的文档格式——
   不适合作为默认模式。
4. `--split-by-pattern` 作为逃生舱仍有存在必要。

## 决策

重新定义为五种一线模式 + 一个逃生舱：

| 模式 | LLM 成本 | 适用场景 | 产出大纲？ |
|------|:--:|------|:--:|
| `heading` | 零 | Markdown 有清晰标题层级 | ✅ |
| `smart` | 极低（~500 tokens） | 主力，有序号的文档，零配置 | ✅ |
| `deep` | 高（全文） | 兜底，无结构纯文本 | ❌（扁平） |
| `manual` | 零 | 用户自己打标记 | ❌ |
| `none` | 零 | 全文校对 | ❌ |
| `--split-by-pattern <regex>` | 零 | 逃生舱，用户自负责任 | ❌ |

关键变更：
- `smart` 从原「全文 LLM 标签」改为「大纲驱动 LLM 切分」（原 `smart-flash`）。
  这是主力模式，覆盖 95% 的文档场景。
- 原 `smart` 重命名为 `deep`，暗示"不惜工本深挖"，仅作为无结构文档的兜底
  方案。不做 Map-Reduce 升级——成本收益不成比例，且 Manual 模式已覆盖极端场景。
- `heading` 提升为独立一线模式（而非隐藏在 rule 的子模式中），因为按标题
  切分足够通用、零成本、自然产出大纲，值得一个明确的 CLI 入口。
- `rule` 从一线模式退为 `--split-by-pattern` 逃生舱，不暴露在 `--mode` 的
  help 文本中。用户需要时主动指定，出问题自己负责。
- 默认模式为 `smart`——零配置、极低成本的主力选项。

## 后果

**正面**:
- 默认 `smart` 模式零配置即可工作，降低了新用户的使用门槛。
- 成本分层清晰：用户知道 `heading` 是免费的、`smart` 几乎免费、`deep` 要花钱。
- 产出大纲的模式（`heading`、`smart`）与后端的结构审查和内容审查自然衔接。
- `deep` 作为兜底存在但不鼓励，避免了 Map-Reduce 等复杂方案的成本陷阱。

**负面**:
- 现有 JiaoDuiAgent 用户习惯的 `rule` 模式不再是 CLI 的一线命令，
  迁移需要调整工作流。
- `smart` 和 `deep` 的命名变更可能造成混淆——文档中明确标注即可缓解。
- `heading` 模式对文档的 Markdown 格式有要求；如果标题写法不规范
  （如用 `**粗体**` 替代 `#`），会被漏过。
