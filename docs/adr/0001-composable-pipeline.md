# ADR-0001: 可组合管线架构

- **状态**: 已采纳
- **日期**: 2026-07-02

## 背景

校对流程包含多个阶段：文档格式转换（如 DOCX → Markdown）、内容拆分（将长文档拆分为可管理的片段）、校对（LLM 驱动的校对）、排版（生成最终输出如 LaTeX）、编译（生成 PDF）。不同的使用场景需要不同的起点和终点。例如，用户可能已有 Markdown 文件，只需从校对阶段开始；也可能需要完整的从 DOCX 到 PDF 的全流程。如果管线是整体耦合的，则每个场景都需要维护一套独立的流程，导致代码重复和维护困难。

## 决策

采用**可组合管线架构**：

- 每个阶段被实现为一个独立可调用的模块，接受标准输入、产出标准输出。
- 阶段之间通过中间文件（如 Markdown、JSON）传递数据。
- CLI 中每个命令对应一个阶段（如 `convert`、`split`、`proofread`、`typeset`、`compile`），用户可以按需组合调用。
- 各阶段产物格式有明确的规范约定，上游输出即下游输入。

## 后果

**正面**:
- 管线高度灵活，用户可根据场景自由组合阶段。
- 每个阶段可独立测试、独立迭代，降低耦合。
- 便于未来插入新阶段或替换现有阶段实现。

**负面**:
- 需要确保上游产物的格式兼容性，若某阶段输出格式变更，可能影响所有下游阶段。
- 中间文件管理增加了用户的认知负担（需了解各阶段的输入输出关系）。
- 跨阶段的全局优化（如增量处理）实现难度增大。
