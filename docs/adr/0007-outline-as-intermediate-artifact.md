# ADR-0007: 大纲作为独立中间产物

- **状态**: 已采纳
- **日期**: 2026-07-02

## 背景

Proseproof v0.2.0 引入了三个需要「文档结构信息」的功能：

1. **Smart 分割**：需要大纲来指导 LLM 做出切分决策
2. **结构审查**：需要在大纲上检查章节顺序、编号连续性、层级一致性
3. **内容审查（Light）**：需要大纲 + 片段摘要来做全局语义矛盾检测

如果每个功能各自提取一遍文档结构，则存在三重冗余：多次扫描同一文档、
各自维护不同的结构表示、不一致风险。同时，Smart 分割产出的结构信息对
后续审查阶段有复用价值——分割本身就是在"理解结构"。

## 决策

**将「大纲」（Outline）提升为 Pipeline 中的独立中间产物**：

- Split 阶段执行时，Python 扫描文档，提取标题（`# ~ ######`）、编号项
  （`1.`、`(1)`、`一、` 等）、列表标记，构建树形大纲结构，落盘为
  `output/{doc}/_outline.json`。
- 大纲条目 `OutlineItem` 包含层级深度、条目类型、纯文本内容、在原文档中
  的行号范围、递归子节点。
- 三个消费者——Smart 分割、结构审查、内容审查——均读取同一份 `_outline.json`，
  不再各自提取。
- 默认提取深度为四级标题（`max_depth: 4`），可通过配置调整。
- 大纲提取的时间点：Split 阶段开始时。如果 Split 为 `none` 或 `manual`
  模式（不产出大纲），则结构审查和内容审查相关步骤自然跳过。

## 后果

**正面**:
- 消除三重冗余提取，三个功能模块的数据一致性由单一数据源保证。
- 大纲落盘为 JSON，用户可手动查看、修正后重跑，增强了可干预性。
- 大纲提取独立于具体分割模式——即使使用 `heading` 模式（零 LLM 成本），
  也能自然产出大纲供后续审查使用。
- Smart 分割现在只需要大纲（~500 tokens）而非全文（~80K tokens），成本
  大幅降低。

**负面**:
- 大纲提取逻辑需要覆盖多种编号格式（中文数字、阿拉伯数字、全角半角、括号
  圆圈），维护成本高于简单的正则切分。
- 大纲准确性决定了 Smart 分割和审查阶段的质量——如果 Python 提取出错
  （如将正文中的 `1.` 误识别为编号），错误会传播到下游。
- 大纲作为共享中间产物，其 schema 变更会影响三个消费者，需要谨慎设计。
